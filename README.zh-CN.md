## 📌 项目简介

本项目为学术论文初稿提供**一套完整的质量保障流水线**。它改编自 [Academic Research Skills](https://github.com/Imbad0202/academic-research-skills) (v3.8.2) 项目，将散装审阅策略系统化为可重复执行的标准流程。

**核心理念：** AI 是审稿人的助理，而非作者。目标是帮人类审稿人抓出那些一眼就能看穿的问题——伪造的参考文献、逻辑断层、方法学漏洞——而不是替作者重写论文。

### 核心功能一览

| # | 功能 | 说明 |
|---|------|------|
| ✅ | **6 种操作模式** | `full` 完整审阅 / `re-review` 验证修改 / `quick` 快速评估 / `methodology-focus` 方法论聚焦 / `guided` 苏格拉底引导 / `calibration` 校准测量 |
| ✅ | **Phase 1** | **多视角同行评审** — 5 个审稿人角色（含独立的魔鬼代言人挑战报告格式） |
| ✅ | **Phase 2** | **编辑决策合成** — 加权打分、决策信函、修改路线图 |
| ✅ | **Phase 2.5** | **修订辅导** — 苏格拉底式对话引导用户自主制定修改策略 |
| ✅ | **Phase 3** | **参考文献完整性验证** — 联网抽查 + 7 项完整性检查清单 |
| ✅ | **Phase 4** | **声明可信度审计**（可选）— 拿原始文献比对论文的核心声明 |

## 🧠 核心逻辑流（长链推理 + 多 Agent 协作）

整个流水线按 **4 个阶段顺序执行**，每个阶段依赖前序结果，形成完整的长链推理链路：

```
Phase 0: 配置 → 读取论文，识别学科/方法/层次，配置 5 位审稿人画像
    ↓（用户确认关口）
Phase 1: 多视角评审 → 5 份结构化审阅报告
    ↓
Phase 2: 编辑合成 → 加权评分 + 决策信 + 修改路线图
    ↓
Phase 3: 完整性门控 → 联网查参考文献是否存在 + 7 项完整性检查
    ↓
Phase 4 (可选): 声明审计 → 联网获取原始文献，比对论文声明的忠实度
```

### 多 Agent 协作机制

**5 个独立审稿人角色**同时/顺序审阅同一篇论文，每个角色独立推理、独立打分：

| # | 角色 | 权重 | 审阅焦点 |
|---|------|------|----------|
| 1 | **主编 (Editor-in-Chief)** | 25% | 领域契合度、创新性、整体质量、写作规范 |
| 2 | **方法论审稿人 (Methodology)** | 25% | 研究设计、统计方法、可复现性 |
| 3 | **领域专家 (Domain)** | 20% | 文献覆盖、理论基础、与相关工作区分度 |
| 4 | **跨学科视角 (Perspective)** | 15% | 跨领域意义、伦理考量、社会影响 |
| 5 | **魔鬼代言人 (Devil's Advocate)** | 15% | 挑战核心假设、识别逻辑谬误、对抗性反论证 |

每个审稿人输出：
- 0-100 评分 + 接受/小修/大修/拒稿 判定
- 具体强弱项（附原文定位）
- 逐条行级批注（位置 → 问题 → 严重程度 → 建议修改）
- 5 维度详细评分矩阵

### 长链推理环节

1. **引用验证（Phase 3）**：对参考文献按 30% 比例抽样（至少 10 篇），逐条通过联网搜索验证文献真实存在性，记录置信度与差异。如果发现伪造引用，自动标记 FAIL 并列出证据。
2. **7 项完整性检查**：数据一致性（p 值是否与显著性匹配）、逻辑一致性（摘要与讨论是否矛盾）、方法对齐（研究方法是否能回答问题）、图表一致性、 disclosures 完整性等。
3. **声明审计（Phase 4）**：从摘要和结论中提取 3-5 个关键声明 → 联网获取被引原文/摘要 → 逐条比对 → 判断忠实度（SUPPORTED / PARTIAL / NOT-SUPPORTED / FABRICATED）。

## 🚀 使用方法

### 作为 Hermes Agent 技能（推荐）

```bash
cp SKILL.md ~/.hermes/skills/research/academic-paper-review/
```

然后在对话中触发：
- "审一下我的论文" / "review my paper"
- "帮我检查引用" / "帮查一下引用是否有问题"
- "run integrity check on this paper"

### 作为独立协议文档

`SKILL.md` 本身是一份自包含的流程协议文档，可以提供给任意 LLM 驱动的编程助手（Claude Code、Codex CLI 等）作为系统提示词。直接将 `SKILL.md` 的内容作为上下文传入即可。

## 📊 审阅报告示例

### 每份审稿人报告结构

| 板块 | 内容 |
|------|------|
| **评分与判定** | 0-100 分，接受/小修/大修/拒稿 |
| **摘要** | 2-3 句总体印象 |
| **优势** | 具体的论文优点，附内容定位 |
| **劣势** | 具体的论文缺点，附证据 |
| **逐条批注** | 位置 → 问题 → 严重程度（Critical/Major/Minor）→ 建议 |
| **详细评分** | 5 项标准 × 0-100 分 + 评语 |

### 编辑决策包

- **决策信函** — 以正式期刊编辑口吻写给作者
- **跨审分析** — 审稿人之间的共识与分歧
- **关键问题** — 被标记为 Critical 的问题汇总
- **修改路线图** — 按优先级（P1/P2/P3）排列的待修项

## ✅ 完整性检查清单（7 项）

| 编号 | 检查项 | 验证内容 |
|------|--------|----------|
| REF-1 | 参考文献存在性 | 抽样的引用文献是否真实存在？ |
| REF-2 | 引用相关性 | 论文对文献的声称与原文献是否一致？ |
| REF-3 | 数据完整性 | 数值结果是否自洽（p 值、样本量、百分比）？ |
| REF-4 | 逻辑一致性 | 论证是否连贯，各章节之间有无矛盾？ |
| REF-5 | 方法对齐性 | 所使用的方法是否真的回答了研究问题？ |
| REF-6 | 图表准确性 | 图表是否与正文描述一致？ |
| REF-7 | 披露完整性 | 局限性、利益冲突、基金来源是否已声明？ |

## ⚠️ 常见陷阱

1. **不可跳过 Phase 0 的用户确认。** 必须展示审稿人配置并让用户确认后才能进入 Phase 1。
2. **审阅前必须通读全文。** 包括摘要、所有章节、图表标题和参考文献。
3. **魔鬼代言人不能太客气。** 必须真正挑战核心假设。如果所有审稿意见都是正面且相似的，那就是失败。
4. **声明审计必须真正读取原文。** 访问不到就标注 UNVERIFIABLE，不可根据论文自己的描述来推断原文内容。
5. **抽样数量要够。** 至少抽 10 篇；50+ 参考文献的论文至少抽 20 篇。
6. **长论文不要用并行模式。** > 3000 词的论文必须用顺序模式，否则子 Agent 上下文不够，产出的审阅会很肤浅。
7. **审完要保存报告。** 所有阶段完成后将最终报告保存到文件。
